When Nuxt loads **`@owdproject/core`**, the engine runs roughly this sequence:

1. **Resolve config path** — **`desktop.config.ts`**, or legacy **`owd.config.ts`**, from the desktop **`rootDir`**.
2. **Validate** — **`theme`**, **`apps`**, **`modules`** shapes; reject Nuxt-option keys mistakenly placed in desktop config.
3. **Split config** — **`splitDesktopUserConfig`**: installer keys vs shell keys. Shell keys target **`runtimeConfig.public.desktop`** and **`appConfig.desktop`** only (never **`_nuxt.options`**).
4. **Initialize runtime desktop** — merge shell config with **defu**; set **`coreVersion`** from the core package.
5. **Install Pinia** — required before theme/modules/apps that use stores or **`defineDesktopApp`** queues.
6. **`installModule` in order:**
   - **`theme`**
   - each entry in **`modules`**
   - each entry in **`apps`**
7. **Shared stack** — PrimeVue, Tailwind, fonts, icon, VueUse, i18n, etc. (as configured by core).
8. **Register globals** — kernel Vue components, composables, stores; client plugins (resize, app registration flush).

If step 1–3 fails, bootstrap **stops** (fail-fast).

## Theme module hook

When the theme’s **`setup`** runs, core has already written user shell config. The theme typically:

```ts
import { defu } from 'defu'

nuxt.options.runtimeConfig.public.desktop = defu(
  nuxt.options.runtimeConfig.public.desktop ?? {},
  options, // module defaults
)
```

User values win; theme **`defaults`** fill gaps.

## Conditional explorer

Themes often gate extra installs:

```ts
if (nuxt.options.modules.includes('@owdproject/module-fs')) {
  await installModule('@owdproject/kit-explorer')
  // theme explorer plugins / components
}
```
